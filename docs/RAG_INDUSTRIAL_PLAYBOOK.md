# 工业版 RAG 学习开发计划

## 1. 这份文档解决什么问题

这份文档记录个人知识库 RAG 从教学版走向工业版的开发顺序、取舍原因、底层逻辑和验证规则。

以后每做一个优化，都要沉淀到这里。代码告诉我们“现在怎么实现”，这份文档告诉我们“为什么先做这个、它解决什么问题、有什么副作用、怎么验证它真的变好”。

使用方式：

1. 先看“阶段路线”，知道下一步该做什么。
2. 再看“已完成决策记录”，理解前面为什么这么做。
3. 每次新优化完成后，在“模块决策模板”里追加一条记录。
4. 遇到检索失败时，按“问题分析规则”记录根因，不写特殊情况补丁。

## 2. 工业版 RAG 的底层链路

工业版 RAG 不是一个功能，而是一条可以被诊断和优化的流水线：

```text
文档解析
-> 文本清洗
-> 切片
-> 元数据增强
-> embedding
-> 向量召回
-> 关键词召回
-> 候选合并
-> rerank
-> 证据过滤
-> 上下文压缩
-> AI 生成
-> 引用校验
-> trace 记录
-> 评测和反馈
```

底层理解：

- 文档解析决定“资料有没有进来”。
- 切片决定“知识颗粒是否合适”。
- 召回决定“正确资料有没有被找到”。
- 排序决定“正确资料有没有排到前面”。
- 证据过滤决定“资料不够时会不会硬答”。
- 生成决定“AI 是否只根据证据说话”。
- Trace 和评测决定“系统能不能持续变好”。

## 4. 已完成决策记录

### 4.1 主链路 MVP

先做什么：

1. 文档扫描。
2. 文档解析。
3. 文本切片。
4. embedding。
5. 向量检索。
6. AI 回答。
7. 引用来源。

为什么先做：

- 先跑通最小闭环，才能知道真实问题在哪里。
- 如果一开始就做 rerank、OCR、复杂评测，很容易不知道问题来自哪一层。

好处：

- 快速形成一个可用产品。
- 可以用真实文档和真实问题测试。

缺点：

- 初版检索会不稳定。
- 切片、召回、排序、回答质量都比较粗。

规则：

- 第一版不做登录、多用户、上传、PDF OCR。
- 先支持 `.md/.docx/.xlsx`，覆盖你的主要资料。
- 答案必须带引用。

验证：

- `POST /api/index/run` 能入库文档。
- `POST /api/search` 能返回 chunks。
- `POST /api/chat` 能返回答案和引用。

### 4.2 Chunk Header / Context Enriched / Document Augmentation

先做什么：

1. 给 chunk 推断 `chunk_header`。
2. 给 chunk 保存上文和下文窗口。
3. 给 chunk 生成潜在问题。
4. embedding 文本中加入文档名、章节、潜在问题、正文。

为什么这样做：

- 原始 chunk 只有正文，像一张被剪下来的纸条。
- 标题和前后文能让系统知道这段话属于哪里。
- 潜在问题让“用户问题”和“文档原文”之间多一座桥。

好处：

- 成本低，不需要新模型。
- 对教学项目很友好，容易理解每个字段的作用。

缺点：

- 规则生成的问题比较粗。
- chunk header 依赖文本格式，Word 样式标题还没真正解析。

具体规则：

- `chunk_header` 优先取 Markdown 标题、章节编号、短标题行。
- 找不到标题时回退到文件名。
- `previous_context` 和 `next_context` 只取相邻 chunk 的短窗口。
- 生成问题只进入 embedding，不当成原文证据展示。

验证：

- `test_chunking.py` 检查切片 overlap、标题和潜在问题。
- 检索结果能看到更清晰的文档标题和 chunk 来源。

### 4.3 Hybrid Search / Rerank v0

先做什么：

1. 保留向量召回。
2. 增加关键词召回。
3. 合并候选。
4. 用轻量公式 rerank。

为什么这样做：

- 向量检索擅长找意思相近的内容。
- 关键词检索擅长找专有名词、字段名、日期、编号。
- 工业 RAG 不能只靠一种召回方式。

好处：

- 专有名词问题更稳。
- 召回漏掉正确片段的概率降低。

缺点：

- 当前关键词召回还是轻量扫描，不是真正 BM25。
- rerank 权重是经验值，需要评测集校准。

当前公式：

```text
最终分 = 向量分 * 0.56
      + 关键词分 * 0.24
      + 标题分 * 0.08
      + 混合召回加分
      + 反馈加分
```

具体规则：

- 同一个 chunk 被向量和关键词同时召回时，标记为 `hybrid`。
- 反馈加分最多正负 `0.12`，避免一次误点影响太大。
- 最终按 rerank 后分数排序。

验证：

- `backend/scripts/eval_retrieval.py` 固定问题通过。
- 检索结果显示召回方式、关键词分、反馈分。

### 4.4 Feedback Loop v0

先做什么：

1. 前端给引用和检索结果加点赞/点踩。
2. 后端新增 `feedback` 表。
3. rerank 读取反馈分，做小幅加权。

为什么这样做：

- 用户反馈是工业 RAG 里最重要的真实信号之一。
- 但反馈不能一开始就强影响结果，先作为轻量权重。

好处：

- 系统可以开始积累真实使用数据。
- 后续可以分析哪些 chunk 经常有帮助。

缺点：

- 当前反馈是按 chunk 全局加权，还没有按问题类型区分。
- 没有反馈备注，暂时不知道用户为什么点踩。

具体规则：

- 点赞记 `+1`，点踩记 `-1`。
- rerank 中反馈分只做小幅加权。
- 不因为单次反馈完全改写排序。

验证：

- `test_feedback.py` 检查正负反馈汇总。
- 前端点反馈后再次检索会参考反馈分。

### 4.5 Field-aware Retrieval

问题来源：

- 用户问“周年庆是哪个游戏的？”
- 文档里确实有答案，但答案在需求基础信息表格字段里。
- 正文 chunk 讲需求内容，表格字段和正文没有稳定关系。

根因：

- 切片后，文档级字段和正文被拆散。
- 系统不知道“项目/部门所属”这种字段属于整篇文档。

先做什么：

1. 入库时提取文档级字段。
2. 每个 chunk 保存 `document_facts`、`chunk_facts`、`field_facts`。
3. embedding、关键词召回、rerank、prompt 都能看到字段。
4. 字段类问题才允许字段证据参与 Self-RAG 判断。

好处：

- “属于哪个项目、负责人是谁、什么时候完成、域名是什么”这类问题更稳。
- 不是写死某个项目，而是补齐字段和正文的关系。

缺点：

- 当前字段抽取还是规则式。
- 表格结构没有完整保留，只是提取了键值对。

具体规则：

- 不写“周年庆=剑网3”这种硬编码。
- 字段标签必须短，不能像正文句子。
- 字段证据不能无条件绕过阈值。
- 只有字段类问题才使用字段证据补充判断。

验证：

- “周年庆是哪个游戏的？”必须命中需求文档。
- “火星移民预算是多少？”仍然必须拒答。

### 4.6 Self-RAG v2

先做什么：

1. 设置最低证据阈值。
2. 第一次检索证据不足时，不马上拒答。
3. 用原问题轻改写和扩大召回做二次检索。
4. 二次检索仍不足才拒答。
5. 前端展示 Self-RAG 状态。

为什么这样做：

- 工业 RAG 既不能硬答，也不能太早拒答。
- 低分时应该先尝试补救，但补救不能把问题改偏。

好处：

- 避免第一次检索偶然漏召回。
- 保持无证据时拒答的安全边界。

缺点：

- 当前补救 query 是规则生成，不是 LLM 改写。
- 还不是完整多步自问、自评、过滤的 Self-RAG。

具体规则：

- 补救只基于原问题和轻量 query variants。
- 不使用低分候选文档标题和摘要改写问题，避免越补越偏。
- 补救后仍低于阈值时必须拒答。

验证：

- 正常问题能回答并带引用。
- 无依据问题触发补救后仍拒答。
- 评测脚本检查 `self_rag` 状态。

### 4.7 RAG Trace v1

先做什么：

1. 新增 `rag_traces` 表。
2. `/api/chat` 每次回答后写入 trace。
3. 新增 `/api/traces`。
4. 前端展示最近问答诊断。

为什么这样做：

- 工业版第一优先级是可观测。
- 没有 trace，就不知道优化到底影响了哪一层。

好处：

- 每次问题都能看到初始分、最终分、是否补救、是否拒答、引用数量、耗时。
- 后续做 Parent-Child、BM25、Query Rewrite 时有对比依据。

缺点：

- 当前只记录摘要级 trace，还没有保存 topK 候选明细。
- 还没有统计报表。

具体规则：

- 只记录答案预览，不保存超长完整答案。
- 记录去重后的引用标题。
- 记录 `initial_best_score` 和 `final_best_score`。
- 记录 `latency_ms`，避免优化后系统变慢却不知道。

验证：

- `/api/chat` 返回 `trace_id`。
- `/api/traces` 能看到最近记录。
- 评测仍保持 `10/10 passed`。

## 5. 下一步模块：Parent-Child Chunk

为什么下一步做它：

- 我们已经有 Trace，可以观察优化前后变化。
- 当前 Small-to-Big 只是“命中块 + 前后窗口”，不是真正 parent-child。
- 长文档或表格文档里，单个 chunk 命中准，但回答可能缺完整上下文。

目标：

```text
小 chunk 负责召回
parent chunk 负责给 AI 回答
```

计划：

1. 入库时生成 parent chunk 和 child chunk。
2. child chunk 存 embedding，用来精准检索。
3. child 命中后，根据 parent_id 找到更完整上下文。
4. prompt 中优先放 parent context，而不是只放 child 正文。
5. trace 记录是否使用 parent context。

好处：

- 小块检索更准。
- 大块回答更完整。
- 减少“命中了但答不全”的问题。

风险：

- parent 太大会增加 token。
- parent 太小又失去上下文意义。
- 需要 trace 和 eval 观察是否真的提升。

初始规则：

- child chunk 继续保持约 800 字。
- parent chunk 可以按 2 到 3 个相邻 child 合并。
- parent context 进入 prompt 前仍要压缩。
- 引用仍然指向实际命中的 child，避免来源变模糊。

验收：

- 原有评测不能下降。
- 长答案问题引用更集中。
- trace 中耗时不能明显失控。

## 7. 检索问题分析规则

以后遇到“明明文档有，系统答不出来”，不要直接写特殊处理。统一按这个顺序分析：

1. 文档是否被扫描到了。
2. 解析出来的纯文本是否包含答案。
3. 答案是否被切到合适 chunk。
4. chunk 是否有标题、字段、前后文。
5. 向量召回是否找到候选。
6. 关键词召回是否找到候选。
7. rerank 是否把正确候选排前。
8. Self-RAG 是否因为阈值拒掉。
9. prompt 是否把证据交给 AI。
10. AI 答案是否被引用支撑。

对应修复原则：

- 解析问题，修 parser。
- 切片问题，修 chunking。
- 字段关系问题，修 metadata。
- 召回问题，修 embedding、关键词检索或 query rewrite。
- 排序问题，修 rerank。
- 阈值问题，修 Self-RAG 规则。
- 生成问题，修 prompt 或 citation check。

## 8. 工业版优先级原则

优先级从高到低：

1. 可观测性：先能看见问题。
2. 数据质量：先保证资料进来是对的。
3. 召回质量：先保证正确资料能被找到。
4. 排序质量：再保证正确资料排前面。
5. 证据安全：资料不够时拒答。
6. 生成质量：最后再优化表达。

大白话：

```text
先看见
再找全
再排准
再防乱答
最后才让 AI 说得更漂亮
```

## 9. 当前学习重点

你现在要重点理解三件事：

1. RAG 失败不是一个原因，而是一条链路中某一层断了。
2. 工业版优化不是堆模型，而是让每一层可观测、可替换、可评测。
3. 每次优化都要留下规则和评测，否则项目会变成一堆“感觉有效”的补丁。

## 10. 子智能体并行开发规则

并行开发可以加快进度，但前提是每个子智能体的边界清楚。工业版 RAG 不是把任务平均分给多个智能体，而是按链路层级拆开，减少文件冲突和逻辑冲突。

### 10.1 主控和子智能体职责

| 角色 | 负责范围 | 不能做什么 | 必须交付 |
| --- | --- | --- | --- |
| 主控智能体 | 拆任务、定接口、合并代码、跑全量测试、推送、更新总文档 | 不把多个核心检索模块同时硬合并 | 最终实现、验证结果、文档总入口 |
| 后端检索智能体 | Parent-Child Chunk、FTS/BM25、Rerank、Self-RAG 后端逻辑 | 不单独改前端大布局，不绕过评测 | 后端代码、单元测试、模块说明 |
| 前端工作台智能体 | Trace 面板、评测面板、诊断 UI | 不改检索算法 | 页面代码、构建验证、交互说明 |
| 评测智能体 | eval case、评测报告、trace 指标统计 | 不改核心业务逻辑 | 评测脚本、指标说明、前后对比 |
| 文档教学智能体 | Playbook、需求文档、交付说明 | 不凭空写未实现能力 | 学习记录、规则、优劣和验证方式 |
| Review 智能体 | 合并前审查风险、边界、漏测 | 不直接重写主实现 | 风险清单、建议修复点 |

大白话：主控像项目负责人，子智能体像专职同事。一个修发动机，一个做测试跑道，一个写维修手册。最后必须由主控统一验收。

### 10.2 哪些模块能并行

| 模块 | 并行建议 | 原因 |
| --- | --- | --- |
| Parent-Child Chunk | 第一批主线 | 它改切片、索引、上下文选择，是 Small-to-Big 的底座。 |
| SQLite FTS/BM25 | 第二批主线 | 它也改索引和检索，和 Parent-Child 容易冲突，不建议同时改核心代码。 |
| LLM Query Rewrite | 可并行预研 | 可以先做独立模块和测试，等检索底座稳定后再接主链路。 |
| Citation Check | 第一批可并行 | 主要检查答案和引用，和索引层冲突小。 |
| Eval Report | 第一批可并行 | 主要读评测和 trace 数据，不影响主链路，最适合并行。 |

第一批推荐并行：

```text
A 后端检索智能体：Parent-Child Chunk
B 评测智能体：Eval Report
C 安全智能体：Citation Check v0
主控智能体：统一接口、合并代码、跑全量测试、更新文档
```

第二批推荐并行：

```text
A 后端检索智能体：SQLite FTS/BM25
B Query 智能体：LLM Query Rewrite
C 评测智能体：Eval Report 增加对比指标
主控智能体：集成到 Self-RAG 和 Trace
```

### 10.3 不建议同时并行的组合

不建议同时让不同子智能体改这些组合：

```text
Parent-Child Chunk + SQLite FTS/BM25
Parent-Child Chunk + 大改 vector_store
FTS/BM25 + 大改 rerank 公式
LLM Query Rewrite + Self-RAG 阈值大改
```

原因：

- Parent-Child 和 FTS/BM25 都会碰 `chunking.py`、`indexer.py`、`vector_store.py`、`retrieval.py`、`rag.py`。
- 两个模块各自测试通过，不代表合起来通过。
- 检索底座变化太多时，评测分数变好或变坏都很难定位原因。

规则：

- 同一批并行任务里，只允许一个子智能体改核心检索底座。
- 其他子智能体优先做评测、文档、前端诊断、安全检查。
- 如果两个子任务必须碰同一个文件，要由主控先定义接口和改动范围。

### 10.5 合并前检查清单

每个并行模块合并前，主控必须检查：

1. 是否有单元测试。
2. 是否跑过 `backend` 测试。
3. 如果改前端，是否跑过前端 build。
4. 是否更新 eval 或说明为什么不用更新。
5. 是否更新 `RAG_INDUSTRIAL_PLAYBOOK.md`。
6. 是否更新 `RAG_OPTIMIZATION_PLAN.md` 的模块状态。
7. 是否影响已有 10 个评测问题。
8. 是否有密钥或本地数据误提交。
9. 是否有文件冲突或重复实现。
10. 是否能用 trace 观察效果。

### 10.7 当前并行开发计划

下一轮建议启动三个并行方向：

| 并行线 | 模块 | 主要目标 | 文档落点 |
| --- | --- | --- | --- |
| A | Parent-Child Chunk | 小块精准召回，大块完整回答 | 本文件新增 Parent-Child 实施记录 |
| B | Eval Report | 输出命中率、拒答率、补救率、平均耗时 | 本文件新增 Eval Report 实施记录 |
| C | Citation Check v0 | 检查答案是否被引用支撑，先记录不拦截 | 本文件新增 Citation Check 实施记录 |

这三条线的关系：

- Parent-Child 改主链路。
- Eval Report 负责量化变化。
- Citation Check 负责回答安全。

它们能并行，是因为目标不同、文件冲突较少、互相能形成验证闭环。
