# RAG 质量说明

RAG 的 demo 很容易，落地效果不好也很常见。原因是它不是一个单点功能，而是一条链路：任意一环做差，最后答案都会差。

## 1. RAG 链路

当前项目的链路是：

```text
本地文档
→ 解析成纯文本
→ 切成小片段
→ embedding 变成向量
→ 向量库检索相关片段
→ 在线 AI 根据片段回答
→ 前端显示答案和引用来源
```

大白话版本：

- embedding 负责“找资料”。
- 在线 AI 负责“组织语言回答”。
- 引用来源负责“让你能检查它是不是瞎说”。

## 2. 为什么 RAG 经常效果不好

常见问题有五类。

### 文档解析差

Word、Excel、PDF 不是天然给 AI 读的。表格、标题、合并单元格、图片里的字都可能丢失结构。

表现：

- 明明资料里有，检索不到。
- Excel 里的信息被拼得很乱。
- AI 答案缺上下文。

### 切片太粗或太碎

切片太粗，检索命中后塞给 AI 的内容很多，噪声也多。
切片太碎，单个片段缺少上下文，AI 读不懂它在说什么。

当前默认：

```text
chunk_size = 800
overlap = 120
```

这是教学阶段的折中值，不是最终最优值。

### 检索只看语义，不看业务结构

向量检索擅长找“意思相近”的内容，但不天然理解：

- 哪个文档更新
- 哪个章节更权威
- Excel 里哪一列是状态
- PRD 里的目标、非目标、验收标准哪个更重要

所以后面要加 metadata 和筛选条件，而不是只靠 embedding。

### Prompt 约束不够

如果不要求“只根据资料回答”，模型会补常识、猜上下文。

当前项目已经做了基础约束：

```text
只能根据用户提供的资料回答。
如果资料中没有依据，明确回答“文档中没有找到依据”。
回答要在关键结论后标注来源编号。
```

### 没有评测集

没有固定问题集，就很难知道“这次改动到底有没有变好”。RAG 要做稳，必须有一组常用问题反复测。

## 3. 我们的质量路线

不要一上来追求“像成熟产品一样聪明”。先把质量变成可观察、可改进。

### 第一步：先验证检索

目标：不接在线 AI，也能找到正确资料。

测试问题：

```text
专题验收助手的目标用户是谁？
专题验收助手要解决什么问题？
十七周年庆小程序有哪些需求？
专项进度计划表里有哪些阶段？
```

验收标准：

- 右侧检索结果命中正确文件。
- 前 3 条结果里能看到答案依据。
- 如果检索不到，先改解析或切片，不急着改 AI。

### 第二步：再验证回答

目标：AI 不能瞎编，必须基于引用回答。

验收标准：

- 回答后面有引用来源。
- 引用片段里能找到答案依据。
- 文档里没有的内容，应该回答“文档中没有找到依据”。

### 第三步：做小型评测集

后续可以加一个 `eval` 脚本，固定 10 到 20 个问题，每次改切片、模型、prompt 都跑一遍。

评测维度：

- 是否命中正确文档
- 是否命中正确片段
- 回答是否引用来源
- 是否拒答无依据问题

## 4. 当前项目的真实状态

已经完成：

- 本地文档目录扫描
- `.md/.docx/.xlsx` 解析逻辑
- 文本切片
- SQLite 元数据
- 本地 SQLite 向量库接口
- 开源 embedding 接口
- `hash` 首跑模式，用于不下载模型时验证全链路
- OpenAI 兼容问答接口
- 中文前端工作台

还需要跑通：

- 用 `hash` 模式先跑通索引和检索
- 网络可用后切到 `fastembed`
- 首次下载 `BAAI/bge-small-zh-v1.5`
- 用真实问题验证检索质量

## 5. 下一步优先级

优先级 1：让向量检索跑通。
原因：检索是 RAG 的地基。

优先级 2：用 5 个真实问题测检索。
原因：先确认系统找得到资料。

优先级 3：接在线 AI 回答。
原因：AI 回答只是最后一层，检索不准时它会把错误包装得更像真的。

优先级 4：增加评测脚本。
原因：RAG 质量要靠固定问题集持续验证，不能只靠感觉。
