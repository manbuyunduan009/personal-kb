# 个人知识库产品专家路线图

## 1. 当前定位

当前项目已经不是单纯的“文档搜索框”，而是一个正在成型的本地产品知识工作台。

短期定位：

- 让你能把本地需求文档、PRD、进度表、复盘资料放进知识库。
- 能问“某个需求是什么、谁负责、有哪些功能、有没有依据”。
- 能看到引用来源、检索诊断、Self-RAG 状态和 Citation Check 风险。

长期定位：

- 从“问答工具”升级为“产品专家助手”。
- 不只回答文档里写了什么，还要能帮你理解历史、识别变更、分析影响、给出方案。

## 2. 明天验收清单

明天先不用切真实 embedding，也不用补大量办公资料。先验收当前工具链路是否稳定。

### 2.1 后端验收

1. 启动后端，打开 `/api/health`。
2. 确认：
   - 后端可用。
   - SQLite 可用。
   - 向量库可用。
   - `openai_configured = true`。
   - 当前 embedding provider 仍允许是 `hash`。
3. 点击前端“索引文档”。
4. 确认文档数量、chunk 数正常。

### 2.2 检索验收

用这几个问题做手动验收：

1. `专题验收助手的目标用户是谁？`
2. `十七周年庆小程序有哪些需求？`
3. `周年庆是哪个游戏的？`
4. `专项进度计划表里有哪些阶段？`
5. `火星移民方案预算是多少？`

预期：

- 前 4 个能命中对应资料。
- 第 5 个应该拒答，不能硬编。
- 检索结果右侧能显示召回方式、BM25、命中关键词等诊断信息。

### 2.3 AI 回答验收

点击“AI 回答”后确认：

- 回答有引用来源。
- 引用来源能对应到真实文档。
- Self-RAG 面板显示证据状态。
- “引用检查”显示 `引用可支撑`、`支撑偏弱`、`支撑不足` 或 `无需检查`。
- 右侧 RAG 诊断新增一条 trace。

### 2.4 评测验收

在后端目录运行：

```powershell
$env:PYTHONPATH='backend'
backend\.venv\Scripts\python.exe backend\scripts\eval_report.py
```

预期：

- `search_pass_rate` 不低于当前基线。
- `chat_pass_rate` 不低于当前基线。
- `api_failure_count = 0`。
- 报告出现 Citation Check 分布、风险率、平均支撑分。

## 3. 暂缓事项

真实 embedding 先不做，等切到办公电脑再做。

原因：

- 真实 embedding 模型会下载到本机缓存，不适合今晚临时处理。
- 办公资料和真实问题集在办公电脑上更完整。
- 现在先把产品能力路线想清楚，明天验收当前版本即可。

后续切办公电脑时再做：

1. 配置真实 embedding provider。
2. 重新索引办公资料。
3. 用真实问题集跑评测。
4. 对比 hash 和真实 embedding 的命中差异。

## 4. 产品专家助手的核心场景

你真正需要的不是“文档里有没有这句话”，而是这些产品工作能力。

### 4.1 当前需求应该怎么做

用户问题示例：

- `这个新需求应该怎么设计？`
- `这个小程序预约规则应该怎么处理？`
- `这种活动需求历史上有没有类似方案？`

系统应该做：

1. 找到当前需求文档。
2. 找到历史相似需求。
3. 提取历史方案、风险、边界条件。
4. 对比当前需求和历史需求差异。
5. 给出推荐方案，并明确依据来自哪些文档。

输出应该包含：

- 推荐方案。
- 参考历史需求。
- 与历史方案的差异。
- 需要产品确认的问题。
- 验收点和风险点。

### 4.2 当前需求做了哪些改变

用户问题示例：

- `这版需求相比上一版改了什么？`
- `7月1日变更了哪些规则？`
- `这次调整会影响哪些页面和接口？`

系统应该做：

1. 识别同一需求的不同版本。
2. 对比标题、字段、正文、表格、日期、状态。
3. 抽取新增、删除、修改、未变更内容。
4. 按模块归类：入口、预约、授权、票务、导航、后台、接口、数据统计。
5. 输出变更摘要和影响范围。

输出应该包含：

- 变更摘要。
- 新增内容。
- 删除内容。
- 修改内容。
- 受影响页面、接口、数据、验收点。
- 需要同步给研发、设计、测试或运营的事项。

### 4.3 对历史修改有哪些

用户问题示例：

- `这个需求历史上改过几次？`
- `最早版本和现在版本差异是什么？`
- `哪些修改是反复出现的？`

系统应该做：

1. 给每份文档建立版本记录。
2. 按时间线展示需求演进。
3. 把每次修改归因到字段、章节或模块。
4. 识别反复修改的内容，提示它可能是高风险区域。

输出应该包含：

- 需求时间线。
- 每次版本变更摘要。
- 高频变更模块。
- 可能的需求不稳定点。
- 建议补充的规则或验收口径。

### 4.4 最好的方案是什么

用户问题示例：

- `参考历史资料，当前最好的实现方案是什么？`
- `这几个方案哪个更适合？`
- `如果我是产品经理，应该怎么推进？`

系统应该做：

1. 检索当前需求。
2. 检索历史相似需求和已有方案。
3. 提取可复用规则。
4. 从用户体验、研发成本、风险、复用性、验收难度几个维度比较。
5. 给出推荐方案和不推荐方案。

输出应该包含：

- 推荐方案。
- 方案理由。
- 取舍分析。
- 风险和兜底方案。
- 需要确认的问题。
- 建议拆分的任务和验收清单。

## 5. 长期开发阶段

### 阶段 0：当前版本稳定验收

目标：

- 保持当前问答、检索、Trace、Citation Check 稳定。
- 不引入真实 embedding。
- 不大改架构。

要做：

1. 明天按验收清单走一遍。
2. 记录失败问题。
3. 把失败问题归类到解析、切片、召回、重排、Self-RAG、Prompt、生成。

完成标准：

- 固定评测能跑通。
- 手动 5 个问题能得到预期结果。
- 文档和交付说明能看懂。

### 阶段 1：需求版本管理

为什么先做：

- 如果要回答“改了什么”，必须先知道“同一个需求有哪些版本”。
- 没有版本关系，系统只能检索单份文档，不能做历史对比。

要做：

1. 给文档增加 `project_key`、`requirement_key`、`version_label`、`version_date`。
2. 从文件名、标题、表格字段中推断需求归属。
3. 新增版本表：
   - requirement id
   - document id
   - version date
   - source path
   - content hash
4. 前端文档列表支持按项目/需求分组。

好处：

- 可以知道一份文档属于哪个项目、哪个需求。
- 可以把多份历史文档串成时间线。

缺点：

- 自动识别不可能一次完全准确。
- 需要后续提供人工修正入口。

验收：

- 能识别“剑网3十七周年庆”相关文档是一组。
- 能显示同一需求的多个版本。

### 阶段 2：需求变更对比

为什么做：

- 产品工作里最常见的问题不是“有没有”，而是“这次和上次有什么不同”。

要做：

1. 按文档版本做结构化 diff。
2. 支持正文段落 diff。
3. 支持表格字段 diff。
4. 把差异分成：
   - 新增
   - 删除
   - 修改
   - 未变
5. 生成变更摘要。

好处：

- 能快速知道需求变更点。
- 可以减少人工逐页对比。

缺点：

- Word 和 Excel 的结构差异比较难。
- 需要保留更多解析结构，不能只存纯文本。

验收：

- 选择两个版本，能输出变更摘要。
- 能指出变更影响到哪些字段或章节。

### 阶段 3：影响范围分析

为什么做：

- 产品经理真正关心的是“这个改动会影响谁、影响哪里、要通知谁”。

要做：

1. 把需求内容抽成模块：
   - 页面
   - 功能
   - 接口
   - 字段
   - 权限
   - 数据统计
   - 运营配置
2. 建立需求知识图谱的轻量版本。
3. 变更发生后，自动推断影响范围。
4. 给出同步对象：
   - 产品
   - 设计
   - 前端
   - 后端
   - 测试
   - 运营

好处：

- 从“看文档”升级到“帮你做产品判断”。
- 能提前发现遗漏验收点。

缺点：

- 抽取规则需要结合你的真实文档慢慢训练。
- 第一版会偏保守，宁可多提示一些风险。

验收：

- 输入一个变更点，系统能输出受影响模块。
- 输出内容必须带引用来源。

### 阶段 4：历史相似需求检索

为什么做：

- 你想问“历史上有没有类似方案”，这需要比普通 RAG 更强的相似需求检索。

当前状态：

- 需求卡片 v0 已提前落地。
- 当前卡片使用规则抽取，不调用 AI，不依赖真实 embedding。
- 已能输出背景、目标、用户/角色、范围/功能、关键规则、风险点、验收点、待确认问题和下一步动作。

要做：

1. 为每个需求生成需求卡片：
   - 背景
   - 目标
   - 用户
   - 核心规则
   - 页面范围
   - 风险点
   - 验收点
2. 对需求卡片做 embedding。
3. 用户问新需求时，先找相似需求卡片，再找具体文档证据。
4. 输出“相似原因”。

好处：

- 比直接搜原文更适合产品复用。
- 可以沉淀你的历史产品经验。

缺点：

- 需要做需求级摘要，不能只靠 chunk。
- 摘要质量会影响后续推荐质量。

验收：

- 输入一个新需求描述，能返回 3 个相似历史需求。
- 每个相似需求说明相似点和不同点。

### 阶段 5：方案推荐助手

为什么做：

- 这是从“资料检索”到“产品专家”的关键一步。

要做：

1. 基于当前需求和历史相似需求生成方案。
2. 输出多方案对比：
   - 方案 A：复用历史规则
   - 方案 B：轻量调整
   - 方案 C：重新设计
3. 每个方案给出：
   - 用户体验
   - 研发成本
   - 风险
   - 适用条件
   - 不适用条件
4. 最后给出推荐方案。

好处：

- 帮你减少从零想方案的时间。
- 能把历史经验转成可复用决策。

缺点：

- 方案推荐必须保持“建议”定位，不能伪装成确定结论。
- 必须有证据引用和不确定性说明。

验收：

- 系统能基于历史资料给出方案建议。
- 每个关键建议都能追溯到引用。
- 没有依据时明确说“历史资料不足”。

### 阶段 6：产品交付助手

为什么做：

- 产品专家不只给方案，还要帮你落地成任务、验收和风险清单。

要做：

1. 从需求生成任务拆解。
2. 生成验收清单。
3. 生成测试关注点。
4. 生成需要确认的问题。
5. 生成给研发/设计/测试/运营的同步摘要。

好处：

- 直接服务你的日常产品工作。
- 可以把问答结果变成可执行交付物。

缺点：

- 需要和你的团队协作格式对齐。
- 后续可能要支持导出 Markdown、Excel 或 Worktile 格式。

验收：

- 输入一个需求文档，系统能生成一版可用的产品交付清单。
- 清单里的关键内容都有来源。

## 6. 明天之后的推荐顺序

短期不做真实 embedding 的情况下，推荐顺序是：

1. 先验收当前版本。
2. 新增“需求版本管理”设计，不急着实现完整 diff。
3. 做一个最小版“需求卡片”。
4. 做“两个文档版本对比”的后端原型。
5. 前端加“需求视图”：项目 -> 需求 -> 版本 -> 文档。
6. 再做“当前需求 vs 历史相似需求”的问答模式。
7. 最后做方案推荐和交付清单。

为什么这个顺序好：

- 先建立数据结构，再做智能回答。
- 先能看清历史，再让 AI 分析历史。
- 先做可验证的 diff，再做主观性更强的方案推荐。

## 7. 未来产品形态

最终这个工具可以有四个工作模式：

### 7.1 知识库问答

回答“文档里写了什么”。

适合：

- 查需求字段。
- 查规则。
- 查负责人。
- 查排期。

### 7.2 需求变更分析

回答“这次和历史有什么不同”。

适合：

- 版本对比。
- 变更摘要。
- 影响范围。
- 同步清单。

### 7.3 历史方案参考

回答“以前类似需求怎么做过”。

适合：

- 找相似项目。
- 找可复用规则。
- 找历史风险。
- 找验收经验。

### 7.4 产品方案建议

回答“现在最好怎么做”。

适合：

- 方案对比。
- 推荐方案。
- 风险兜底。
- 任务拆解。

## 8. 长期成功标准

这个工具正式变得有价值，不是因为它能聊天，而是因为它能帮你少做重复脑力活。

成功标准：

- 查资料时间减少。
- 对比历史版本时间减少。
- 需求遗漏变少。
- 方案讨论更有依据。
- 新需求能快速找到历史参考。
- 每个建议都有引用，不靠空想。

## 9. 当前不要做的事

为了不把项目做散，以下暂时不做：

- 暂不做多用户登录。
- 暂不做云部署。
- 暂不做真实 embedding 下载和迁移。
- 暂不做 PDF/OCR。
- 暂不做复杂权限系统。
- 暂不直接接企业 IM 或 Worktile。

这些都可以以后做，但不是当前最能提升你产品工作的核心。

## 10. 下一次开工建议

明天建议从这里开始：

1. 先按“明天验收清单”跑一遍。
2. 把不满意的问题记录下来。
3. 如果当前链路稳定，就开始做“需求版本管理”的数据结构。
4. 然后做一个最小版“需求变更对比”原型。

最重要的判断：

- 如果你只是想查资料，继续优化检索。
- 如果你想让它像产品专家，就要开始建设“需求版本、历史方案、变更影响、方案推荐”这四层能力。
